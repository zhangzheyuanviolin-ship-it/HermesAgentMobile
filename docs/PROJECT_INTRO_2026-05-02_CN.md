# HermesAgentMobile 项目介绍

## 项目定位

HermesAgentMobile 是一个面向 Android 设备的独立智能体运行容器与交互前端，目标是在不依赖 Termux、不过度依赖桌面环境、尽可能保留原生 Android 交互体验的前提下，把 Hermes Agent 带到手机上稳定运行。

项目不是单纯的聊天壳，而是一套完整的移动端智能体落地方案。它同时覆盖了 Ubuntu rootfs 启动、Python 运行环境、Hermes Agent 网关、Flutter 原生前端、后台常驻服务、终端会话、模型配置、会话管理、附件接入，以及 Android 系统侧命令执行能力。

## 上游脉络

项目脉络共有三层：

1. 上游基础能力来源于 `mithun50/openclaw-termux`
   它提供了移动端 shell、proot、终端、服务化和 Android 容器化的整体思路。

2. 直接上游是 `Binair-Dev/HermesAgentMobile`
   这个阶段已经把移动容器承载对象从 OpenClaw 转向 Hermes Agent，形成了 Hermes Agent Mobile 的最初形态。

3. 当前仓库 `zhangzheyuanviolin-ship-it/HermesAgentMobile`
   在直接上游基础上继续向“可长期使用的安卓端智能体应用”推进，重点补齐了中文化、无障碍可读性、聊天页稳定性、多模型配置、测试版并行安装链路，以及 Shizuku 系统命令能力。

此外，底层智能体本体来自 `nousresearch/hermes-agent`。换句话说：

- OpenClaw Termux 解决的是“移动端容器承载能力”。
- Hermes Agent 解决的是“智能体运行能力”。
- HermesAgentMobile 解决的是“把智能体稳定地交付到 Android 手机上，并让普通用户能真正使用”。

## 当前版本最有价值的地方

与直接上游相比，当前仓库已经形成了自己的不可替代性，核心不在“能跑起来”，而在“可持续使用”。

### 1. 双链路发布策略

项目明确拆分为两条安装链路：

- 原始正式版：`com.nxg.hermesagentmobile`
- 并行测试版：`com.nxg.hermesagentmobile.test`

这使得高风险改动可以先在测试版验证，再决定是否覆盖正式版，极大降低了移动端持续迭代时的回滚成本和使用风险。

### 2. 原生聊天页已经具备实际可用性

当前聊天页不再只是把 Hermes 输出直接堆给用户，而是按使用场景拆成多个可控区域：

- 用户输入区
- 工具执行过程区
- AI 思考内容区
- 最终回复区

并且支持思考区显示开关、会话持久化、附件接入、滚动管理和多轮上下文恢复。这是从“能看到日志”升级到“能作为日常主界面使用”的关键一步。

### 3. Shizuku 系统命令能力已完成接入

当前版本已经把 Android 系统命令能力接入 Hermes 运行链路：

- 设置页可管理 Shizuku 权限与通道开关
- 原生层提供 `system-shell` 与 `system-shell-status` 包装命令
- 智能体系统提示已注入正确调用方式
- 状态展示不再只看授权，而会校验命令是否真实就绪

这意味着 Hermes 不只是在 Ubuntu 容器里工作，还能安全、可控地触达 Android 系统命令层。

### 4. 状态一致性和服务生命周期被补强

这一轮修复解决了一个非常关键的问题：前端状态、原生权限、后台服务、容器内命令就绪状态必须一致。当前版本已经补上：

- 桥接脚本缺失时自动补写
- 停止网关时清理残留进程
- 重新启动时先修环境再决定是否接管旧实例
- 设置页状态与命令真实可用状态联动

这类改动不会直接体现在界面花哨程度上，但它决定了应用是不是“真的可靠”。

## 当前核心功能

### 宿主与环境层

- Android 原生前台服务常驻 Hermes 网关
- proot Ubuntu rootfs 启动与维护
- Python 运行环境与 Hermes Agent 依赖安装
- 根文件系统目录修复、DNS 写入与环境补丁

### 智能体与交互层

- 原生 Flutter 聊天页
- 会话管理与历史持久化
- 附件接入
- 工具过程、思考内容、最终回复分区展示
- 多模型配置、选择、编辑与探测

### 运维与调试层

- 终端会话
- 网关日志
- 自动启动与电池优化设置
- 存储权限管理
- Shizuku 系统命令通道

## 与上游的差异总结

### 相比 `Binair-Dev/HermesAgentMobile`

当前仓库新增或强化了以下方向：

- 更明确的正式版 / 测试版双包发布策略
- 中文界面与移动端无障碍可读性改进
- 原生聊天页的多区域稳定分流
- 多模型配置管理能力
- 会话与附件链路
- Shizuku 系统命令桥接
- 状态一致性与后台网关生命周期修复

### 相比 `mithun50/openclaw-termux`

当前仓库的重点不再是通用 shell 容器，而是：

- 以 Hermes Agent 为核心对象的专门化集成
- 原生 Android 智能体应用式交互
- 面向最终用户的安装、配置、聊天、发布和回滚链路

### 相比 `nousresearch/hermes-agent`

当前仓库补的是移动端落地层：

- Android 前端
- 本地容器运行时
- 后台服务管理
- 模型配置与本地会话体验
- Android 系统侧能力桥接

## 项目适合的使用场景

- 希望在 Android 手机上长期运行 Hermes Agent 的用户
- 需要同时具备聊天界面、终端能力和后台常驻能力的移动端场景
- 希望让智能体直接操作 Android 系统命令的高级用户
- 需要正式版与测试版并行共存、可灰度升级的持续开发场景

## 当前阶段结论
